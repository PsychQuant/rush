## Section A — v0.1 cleanup (4 minor items, ~1-2 hours total)

These were flagged in the final v0.1.0 review but deferred so v0.1.0 could ship. Address before v0.2 release.

### A1. Fix `TDXError.rateLimited` error message text

**Issue**: Says "retry in 60s" but the actual single-retry sleeps 1s only, then throws if second attempt also 429s. The 60s figure was aspirational (TDX's per-minute reset window) but doesn't match the code path.

**File**: `Sources/CheTransportMCP/TDXClient.swift` — find `case .rateLimited:` in `errorDescription`.

**Fix**: Change `"TDX rate limit exceeded; retry in 60s"` to `"TDX rate limit exceeded after retry; please slow request rate or wait for the per-minute window to reset"` or similar.

### A2. Document `window_min` limitation in CLAUDE.md

**Issue**: `rail_status_station` schema accepts `window_min` integer, but TDX endpoint returns its own default window — the param is currently informational only. An LLM agent passing `window_min: 120` gets back a 60-min window with no explanation.

**File**: `CLAUDE.md` — add a note under the Rail tools section.

**Fix**: Add bullet under `rail_status_station` description:

> Note: `window_min` is accepted for forward-compatibility but currently ignored — TDX `StationLiveBoard` endpoint returns its own default window. Client-side filtering by window is planned for v0.3.

### A3. Parallelize 8-system fetch in `rail_search_stations`

**Issue**: When `system` param is omitted, the executor loops sequentially through 8 systems with 8 sequential HTTP calls. Cold cache = up to 16 seconds end-to-end. Steady-state OK because 24h cache hides this, but UX on first call is poor.

**File**: `Sources/CheTransportMCP/Tools/RailTools.swift` — `executeSearchStations` function.

**Fix**: Replace the `for sys in systemFilter` loop with `withTaskGroup(of:)` to fetch in parallel:

```swift
let perSystemResults = try await withThrowingTaskGroup(of: (RailSystem, [RailStation]).self) { group in
    for sys in systemFilter {
        group.addTask {
            let data = try await client.fetch(
                path: "\(sys.apiPath)/Station",
                cacheTTL: 86400,
                cache: cache
            )
            return (sys, Self.decodeStationList(data: data))
        }
    }
    var collected: [(RailSystem, [RailStation])] = []
    for try await result in group {
        collected.append(result)
    }
    return collected
}
```

Then iterate `perSystemResults` to build `allMatches`. **Caveat**: TDX free tier is 50 req/min — 8 parallel requests is well within that, but if Plan 2's bus tools do the same parallelization across more endpoints, may need a semaphore.

### A4. Add Cache size cap

**Issue**: `Cache` actor is unbounded. For v0.1 (rail stations ~500 KB total) negligible, but v0.2's bus tools will pull thousands of stops per city. Without a cap, long-running MCP sessions could accumulate memory.

**File**: `Sources/CheTransportMCP/Cache.swift`.

**Fix options**:

1. **LRU with hard cap** (e.g., 1000 entries max) — most robust, but needs OrderedSet-like state
2. **Per-key TTL only, no count cap, periodic prune** — simpler, less robust under bursts
3. **Per-entry size estimate + total-size cap** — flexible but more code

Recommend option 1 for v0.2. Will need `private var keyOrder: [String] = []` to track LRU and bump-on-access in `get`.

## Section B — Plan 2 (Bus + Bike tools, ~1 week)

The next planned increment. Write the spec + plan first, then execute via subagent-driven-development.

**Tools to add**:

### Bus (5)

- `bus_search_routes(query, city?)` — route name / number search
- `bus_search_stops(query, city?)` — stop name search
- `bus_find_routes(from_stop, to_stop, city?)` — O/D candidate routes
- `bus_status_arrivals(stop_id, city?)` — near-term arrival predictions at a stop
- `bus_status_positions(route_id, city?)` — live bus positions on a route

### Bike (3)

- `bike_search_stations(query, system?)` — YouBike station search
- `bike_stations_nearby(lat, lon, radius_m, system?)` — geographic nearby + live availability
- `bike_status_station(station_id)` — single-station live availability

**Important constraints**:

- **City scoping for bus**: TDX bus endpoints are city-scoped (`/Bus/Route/City/{City}/{RouteName}`). Need a `BusCity` enum (Taipei, NewTaipei, Taoyuan, Hsinchu, ... ~22 cities). Question: do we fan out across all cities when `city` is omitted (like rail's all-systems search), or require it?
- **YouBike 1.0 vs 2.0**: TDX has separate endpoints. Default to 2.0, accept `system` param to switch.
- **`bus_status_arrivals` doesn't take time** — TDX returns the next few minutes' arrivals from the current moment. Document this in tool description.

**Architecture changes needed**:

- Create `Sources/CheTransportMCP/Tools/BusTools.swift` and `BikeTools.swift` following the same pattern (`defineTools()` + `handleCall` switch case + private executor)
- Create `Sources/CheTransportMCP/Models/BusModels.swift` and `BikeModels.swift`
- Update `Server.swift` to call `BusTools.register(server:client:cache:)` and `BikeTools.register(...)` after `RailTools.register(...)`
- **Risk**: how does `Server.swift` aggregate `defineTools()` from multiple tool modules into the single `ListTools` handler? Need to refactor: each `*Tools.register` should append to a shared `[Tool]` instead of installing its own `ListTools` handler

**Suggested approach for Plan 2**:

1. Refactor: introduce `ToolRegistry` actor or similar that aggregates `[Tool]` from all `*Tools.register` calls, then `Server.swift` installs a single `ListTools` handler reading from it
2. Then add Bus + Bike modules in parallel
3. Same TDD discipline + spec/quality reviews per task
